# 因素11：从任何入口触发智能体

> 来源：12-Factor Agents / Factor 11（Trigger from anywhere, meet users where they are）

## 一句话先懂

智能体不能只待在一个聊天窗口里，要能被 Slack、邮件、Webhook、Cron 等多入口触发，并在用户熟悉渠道反馈。

## 奶奶版理解

家里有事，不会只靠一部电话。  
电话、微信、当面说都可以，关键是“事情不断线”。

## 原文核心价值

- 让用户在原有工作流里使用 AI，而不是切换新平台。
- 支持“外循环任务”：由事件触发，不一定由人发起。
- 在关键节点快速拉人确认，才能做更高价值动作。

## 推荐落地架构

1. **接入层（Adapter）**：Slack/Email/Webhook/Cron 统一入站。
2. **规范事件模型**：source、actor、payload、thread_id。
3. **编排层**：根据事件恢复或启动 agent 线程。
4. **回执层**：把结果回推到原渠道。

## 真实业务场景

- 监控告警触发 agent 自动排查。
- 排查中发现需人工批准，发到值班群。
- 人工回复“同意执行”后继续修复。
- 修复结果回写群里并归档审计。

## 常见误区

- 各渠道协议不统一，行为不一致。
- 用户身份映射缺失，权限容易错乱。
- 没有 thread_id 贯穿，跨渠道上下文断裂。

## 我的思考

很多产品“效果不错但用不起来”，不是模型问题，而是入口问题。  
这条提醒我们：AI 产品不是只做“模型体验”，还要做“工作流体验”。

## 小结

- 多入口是可用性，不是炫技。
- 通道可以多样，协议必须统一。
- 真正可落地的 agent 一定支持跨渠道闭环。
